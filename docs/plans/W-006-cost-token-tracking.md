# W-006: Cost & Token Tracking

## 1. Goal and Scope

### Problem

Each pipeline run invokes multiple Claude agents (planner, coordinator, implementer, tester, guardian) across multiple stages, often with loop-back iterations (test failure -> re-implement, review changes -> re-implement). There is currently no structured way to answer:

- How many tokens did a single run consume, broken down by stage and iteration?
- What was the dollar cost per stage, per agent, per run?
- How do costs trend across runs over time?
- Which stages or loops are the most expensive?

### Goal

Instrument the pipeline to capture, store, and aggregate token usage and cost data at every granularity level: per-iteration, per-stage, per-run, and cumulative across runs. This data will power the W-015 Token Dashboard in the UI.

### Metrics to Track

Per agent invocation (each iteration of each stage):

| Metric | Source | Notes |
|---|---|---|
| `input_tokens` | `result.usage.input_tokens` | Tokens sent to the model |
| `output_tokens` | `result.usage.output_tokens` | Tokens generated by the model |
| `cache_creation_input_tokens` | `result.usage.cache_creation_input_tokens` | Tokens written to prompt cache |
| `cache_read_input_tokens` | `result.usage.cache_read_input_tokens` | Tokens read from prompt cache |
| `total_cost_usd` | `result.total_cost_usd` | CLI-reported cost (authoritative) |
| `duration_ms` | `result.duration_ms` | Wall-clock time for the agent call |
| `num_turns` | `result.num_turns` | Number of conversational turns |
| `model` | `result.model` or stage config | Which model was used |
| `agent` | Stage config | Which agent definition was used |

Derived metrics (computed, not stored raw):

- `total_tokens` = `input_tokens + output_tokens`
- `cost_per_1k_tokens` = `total_cost_usd / (total_tokens / 1000)`
- `cache_hit_rate` = `cache_read_input_tokens / (cache_read_input_tokens + input_tokens)` when applicable

## 3. Claude CLI Output Parsing

### 3.1 Current State

The pipeline already uses `--output-format stream-json --verbose`. The final event in the NDJSON stream has `type: "result"` and contains:

- `total_cost_usd` (float) -- already extracted in `runner.py` line 557-558
- `num_turns` (int) -- already extracted in `runner.py` line 555-556
- `duration_ms` (int) -- already extracted in `runner.py` line 199
- `usage` (object) -- partially referenced in `runner.py` line 214-217 for logging only

The `usage` object in the result event has this structure (from Claude CLI stream-json output):

```json
{
    "input_tokens": 28500,
    "output_tokens": 4200,
    "cache_creation_input_tokens": 12000,
    "cache_read_input_tokens": 8500
}
```

The `model` field is available from the `system.init` event (`event.model`) which is already logged at line 91-92 of `claude_cli.py`.

### 3.2 Extraction Strategy

No new CLI flags needed. All required data is already present in the `raw_envelope` (the result event dict) returned by `run_agent()`. The extraction happens in `runner.py` where `run_stage()` returns `(structured_output, raw_envelope)`.

Fields to extract from `raw_envelope`:

| Field | Path | Fallback |
|---|---|---|
| `input_tokens` | `raw_envelope["usage"]["input_tokens"]` | `0` |
| `output_tokens` | `raw_envelope["usage"]["output_tokens"]` | `0` |
| `cache_creation_input_tokens` | `raw_envelope["usage"]["cache_creation_input_tokens"]` | `0` |
| `cache_read_input_tokens` | `raw_envelope["usage"]["cache_read_input_tokens"]` | `0` |
| `total_cost_usd` | `raw_envelope["total_cost_usd"]` | `0.0` |
| `duration_ms` | `raw_envelope["duration_ms"]` | `0` |
| `num_turns` | `raw_envelope["num_turns"]` | `0` |
| `model` | `raw_envelope["model"]` | stage config model |

### 3.3 Model Name Resolution

The `system.init` event contains the full model identifier (e.g., `claude-sonnet-4-20250514`). To capture this, `process_stream()` should extract and return the model name alongside the result event. Currently the init event model is only logged as a formatted line -- the actual value is discarded.

Change: have `process_stream()` capture the model from the init event and attach it to the result event if not already present.


## 4. Storage Design

### 4.1 Decision: Extend Existing Status/Result JSON (Primary) + Separate Cumulative File

Rationale:
- Per-iteration and per-stage token data belongs inside the status JSON because it is tightly coupled to the run lifecycle and is already partially tracked (`cost_usd`, `turns`).
- Cumulative cross-run stats belong in a separate file because archived run JSONs are immutable snapshots and aggregation requires reading across them.

### 4.2 File Locations

| Data | File | Lifecycle |
|---|---|---|
| Active run token data | `.worca/status.json` | Written during pipeline, archived on completion |
| Archived run token data | `.worca/results/{run_id}.json` | Immutable after archival |
| Cumulative stats | `.worca/stats/cumulative.json` | Updated after each run completes |
| Per-iteration raw envelope | `.worca/logs/{stage}/iter-{N}.json` | Already exists, contains full usage data |

### 4.3 Backward Compatibility

Existing status.json and result files lack `token_usage` fields. The code must handle their absence gracefully:

- All token usage reads use `.get("token_usage", {})` with field-level defaults of `0`.
- The cumulative stats builder can reconstruct partial data from existing archived runs by reading `total_cost_usd` and `num_turns` from stage-level data (already present).
- A one-time migration function can backfill `token_usage` for archived runs by re-reading their `.worca/results/{run_id}/` log directories (iter-N.json files contain the raw envelopes with usage data).


## 5. Pricing Model

### 5.1 Approach: CLI-Reported Cost as Authoritative

The Claude CLI already computes and reports `total_cost_usd` per invocation. This is the authoritative cost figure. We store it directly rather than computing cost from tokens.

### 5.2 Reference Pricing (for Validation and Estimation)

Store a reference pricing table in settings for display purposes and offline estimation. This is NOT used for billing -- only for UI display of rate information and estimating costs when CLI cost data is unavailable.

Add to `.claude/settings.json` under `worca.pricing`:

```json
{
    "worca": {
        "pricing": {
            "models": {
                "opus": {
                    "input_per_mtok": 15.00,
                    "output_per_mtok": 75.00,
                    "cache_write_per_mtok": 18.75,
                    "cache_read_per_mtok": 1.50
                },
                "sonnet": {
                    "input_per_mtok": 3.00,
                    "output_per_mtok": 15.00,
                    "cache_write_per_mtok": 3.75,
                    "cache_read_per_mtok": 0.30
                }
            },
            "currency": "USD",
            "last_updated": "2025-05-01"
        }
    }
}
```

### 5.3 Cost Estimation Function

A utility function computes estimated cost from token counts and the pricing table. Used only when `total_cost_usd` is missing (e.g., interrupted runs, backfill of old data). Formula:

```
estimated_cost = (input_tokens * input_per_mtok / 1_000_000)
               + (output_tokens * output_per_mtok / 1_000_000)
               + (cache_creation_input_tokens * cache_write_per_mtok / 1_000_000)
               + (cache_read_input_tokens * cache_read_per_mtok / 1_000_000)
```


## 6. Aggregation

### 6.1 Stage-Level Aggregation

After each iteration completes, recompute the stage-level aggregate by summing across all iterations for that stage. This happens inline in `runner.py` after `complete_iteration()`.

### 6.2 Run-Level Aggregation

When the pipeline completes (or is archived), compute the run-level aggregate by summing across all stage aggregates. Add `by_model` and `by_stage` breakdowns. This happens in `run_pipeline()` right before the final `save_status()`.

### 6.3 Cumulative Aggregation

After `_archive_run()` moves a completed run to results, update `.worca/stats/cumulative.json`:

1. Read existing cumulative stats (or initialize empty).
2. Add the new run's aggregate data.
3. Recompute totals.
4. Append a summary entry to the `runs` array.
5. Write back.

### 6.4 Rebuild Command

Provide a CLI utility to rebuild cumulative stats from all archived runs:

```bash
python -m worca.utils.rebuild_stats
```

This reads every `.worca/results/*.json`, extracts token_usage, and rebuilds `cumulative.json` from scratch. Useful after manual edits, migrations, or data corruption.
